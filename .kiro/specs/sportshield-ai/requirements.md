# Requirements Document

## Introduction

SportShield AI is a production-grade, full-stack anti-piracy and unauthorized sports media tracking platform targeting sports organizations (IPL, FIFA, NBA, broadcasters, leagues). The system authenticates official media assets, detects unauthorized usage across internet uploads, tracks stolen or edited copies in near real time, flags suspicious propagation anomalies, and surfaces alerts and analytics through a futuristic, investor-demo-ready dashboard.

The MVP is designed for hackathon demonstration with mock crawlers and simulated data, but the architecture must be modular and scalable to support production deployment.

---

## Glossary

- **System**: The SportShield AI platform as a whole
- **Asset**: An official media file (video, image, logo, or broadcast clip) registered by an organization
- **Fingerprint**: A perceptual hash or embedding vector derived from an Asset that uniquely identifies its visual content
- **Embedding**: A high-dimensional vector representation of an Asset generated by a CNN model (ResNet or CLIP)
- **FAISS_Index**: The Facebook AI Similarity Search index used to store and query Embeddings
- **Scan_Job**: An asynchronous background task that compares a suspicious upload against the FAISS_Index
- **Incident**: A detected match between a suspicious upload and a registered Asset that exceeds the similarity threshold
- **Match_Score**: A floating-point confidence value between 0.0 and 1.0 representing similarity between two media items
- **Threshold**: The configurable minimum Match_Score above which a comparison is flagged as an Incident
- **Crawler**: A mock service that simulates ingestion of suspicious media from external sources (YouTube, websites, social media)
- **Certificate**: A mock blockchain-style authenticity token generated for each registered Asset
- **Organization**: A sports rights holder (league, broadcaster, club) that registers Assets on the platform
- **Admin**: An authenticated user with full access to all platform features
- **Forensics_Viewer**: The UI component that renders side-by-side comparison of an original Asset and a suspicious upload
- **Alert_Dashboard**: The real-time UI panel displaying live Incident feeds, statistics, and charts
- **Anomaly**: A suspicious pattern in Incident propagation that deviates from expected distribution

## Requirements

### Requirement 1: Official Media Asset Registration

**User Story:** As an Organization, I want to upload and register official media assets, so that the system can protect them against unauthorized use.

#### Acceptance Criteria

1. THE System SHALL accept file uploads for the following media types: MP4, MOV, AVI (video); JPG, PNG, WebP (image); SVG, PNG (logo); and MP4, MOV (broadcast clips).
2. WHEN an Asset is uploaded, THE System SHALL generate a perceptual Fingerprint for the Asset within 30 seconds of upload completion.
3. WHEN an Asset is uploaded, THE System SHALL extract an Embedding vector and store it in the FAISS_Index within 60 seconds of upload completion.
4. WHEN an Asset is uploaded, THE System SHALL store the following metadata: organization name, asset title, asset type, upload timestamp, file size, and duration (for video assets).
5. THE System SHALL support drag-and-drop single file upload via the registration UI.
6. THE System SHALL support bulk upload of up to 50 files in a single batch operation.
7. WHEN a bulk upload is initiated, THE System SHALL display per-file progress indicators showing upload percentage and processing status.
8. WHEN an Asset upload completes, THE System SHALL display a thumbnail preview for image assets and a keyframe preview for video assets.
9. IF an uploaded file exceeds 500 MB, THEN THE System SHALL reject the upload and return a descriptive error message specifying the size limit.
10. IF an uploaded file format is not in the supported list, THEN THE System SHALL reject the upload and return a descriptive error message listing accepted formats.
11. WHEN an Asset is successfully registered, THE System SHALL generate a Certificate containing: asset ID, organization name, upload timestamp, Fingerprint hash, and a mock blockchain transaction hash.
12. THE System SHALL display the Certificate on the asset detail page with a visual authenticity badge.

### Requirement 2: AI Similarity Detection Engine

**User Story:** As an Organization, I want the system to detect visually similar or altered copies of my registered assets, so that I can identify unauthorized usage even when content has been modified.

#### Acceptance Criteria

1. WHEN a video Asset is registered, THE System SHALL extract keyframes at a configurable interval (default: every 5 seconds) and generate an Embedding for each keyframe.
2. WHEN an image Asset is registered, THE System SHALL generate both a perceptual hash and a CNN Embedding for the Asset.
3. WHEN a Scan_Job compares a suspicious upload against the FAISS_Index, THE System SHALL return a Match_Score between 0.0 and 1.0 for the closest matching Asset.
4. THE System SHALL detect altered copies including: cropped versions, resized versions, watermark-removed versions, recompressed versions, and versions with minor color or brightness edits.
5. WHEN a Match_Score meets or exceeds the configured Threshold, THE System SHALL create an Incident record containing: suspicious upload URL, matched Asset ID, Match_Score, detection timestamp, and source type.
6. THE System SHALL support a configurable Threshold value between 0.50 and 0.99, with a default value of 0.80.
7. WHEN an Incident is created, THE System SHALL generate a similarity heatmap image highlighting the regions of visual overlap between the suspicious upload and the matched Asset.
8. FOR ALL registered Assets, querying the FAISS_Index with an Embedding derived from the same Asset SHALL return a Match_Score of 0.95 or higher (self-similarity invariant).
9. THE System SHALL complete a single-asset similarity query against an index of up to 10,000 Embeddings within 500 milliseconds.

### Requirement 3: Unauthorized Scan Monitor

**User Story:** As an Organization, I want the system to continuously scan for unauthorized copies of my assets across simulated internet sources, so that I receive timely detection without manual effort.

#### Acceptance Criteria

1. THE System SHALL include a mock Crawler that simulates ingestion of suspicious media from three source types: YouTube upload simulation, website scraped media simulation, and social media feed simulation.
2. WHEN the Scan Monitor is active, THE System SHALL execute Scan_Jobs at a configurable interval (default: every 60 seconds).
3. WHEN a Scan_Job is dispatched, THE System SHALL process it asynchronously without blocking the API or UI.
4. THE System SHALL maintain a Scan_Job queue that supports at least 100 concurrent pending jobs.
5. WHEN a Scan_Job completes and a match is found above the Threshold, THE System SHALL create an Incident and push a real-time notification to the Alert_Dashboard within 5 seconds.
6. THE System SHALL generate demo data including at least 50 simulated Incidents with varied Match_Scores, source types, timestamps spanning the past 30 days, and geographic origin metadata.
7. WHEN the demo data generator is triggered, THE System SHALL populate the database with the simulated Incidents within 10 seconds.
8. IF a Scan_Job fails due to an internal error, THEN THE System SHALL log the error, mark the job as failed, and retry the job up to 3 times before marking it as permanently failed.

### Requirement 4: Real-Time Alert Dashboard

**User Story:** As an Admin, I want a live dashboard showing piracy statistics and new incidents as they are detected, so that I can monitor threats and respond quickly.

#### Acceptance Criteria

1. THE Alert_Dashboard SHALL display the following summary panels: total assets protected, total piracy incidents detected, active threats count, and current scan frequency.
2. WHEN a new Incident is created, THE Alert_Dashboard SHALL display it in the live incident feed within 5 seconds without requiring a page refresh.
3. THE Alert_Dashboard SHALL display a time-series chart showing the count of Incidents per day over the past 30 days.
4. THE Alert_Dashboard SHALL display a geographic heatmap showing the simulated origin country of each Incident.
5. THE Alert_Dashboard SHALL display a category breakdown chart showing Incident counts grouped by source type (YouTube, website, social media).
6. WHEN the Alert_Dashboard is loaded, THE System SHALL render all charts and panels within 3 seconds using pre-aggregated data.
7. THE Alert_Dashboard SHALL display each Incident in the live feed with: source URL, matched asset name, Match_Score as a percentage, detection timestamp, and source type icon.
8. WHILE the Alert_Dashboard is open, THE System SHALL poll for new Incidents at a maximum interval of 10 seconds.

### Requirement 5: Media Forensics Viewer

**User Story:** As an Admin, I want to inspect a detected incident in detail with side-by-side media comparison, so that I can assess the severity of the infringement and gather evidence.

#### Acceptance Criteria

1. WHEN an Incident is selected from the Alert_Dashboard or incident list, THE Forensics_Viewer SHALL open and display the original Asset alongside the suspicious upload.
2. THE Forensics_Viewer SHALL provide an interactive slider that reveals the original Asset on one side and the suspicious upload on the other side simultaneously.
3. THE Forensics_Viewer SHALL display the similarity heatmap overlay on the suspicious upload, highlighting regions of visual similarity.
4. THE Forensics_Viewer SHALL display the following Incident metadata: Match_Score as a percentage, detection timestamp, source URL, source type, and asset title.
5. THE Forensics_Viewer SHALL display tampering indicators identifying which modification types were detected (crop, resize, watermark removal, recompression, color edit).
6. WHEN the suspicious upload is a video, THE Forensics_Viewer SHALL display synchronized frame-by-frame comparison at the keyframe level.
7. IF the suspicious upload media is unavailable, THEN THE Forensics_Viewer SHALL display a placeholder with the last known thumbnail and a "Media Unavailable" label.

### Requirement 6: Blockchain Authentication Certificate

**User Story:** As an Organization, I want each registered asset to have a tamper-evident authenticity certificate, so that I can prove ownership and registration timestamp in disputes.

#### Acceptance Criteria

1. WHEN an Asset is registered, THE System SHALL generate a Certificate containing a mock blockchain transaction hash derived from the Asset's Fingerprint, organization ID, and upload timestamp using SHA-256.
2. THE System SHALL display the Certificate on the asset detail page showing: transaction hash, block number (mock sequential integer), timestamp, organization name, and asset Fingerprint hash.
3. THE System SHALL display a visual "Verified on Chain" badge alongside the Certificate.
4. FOR ALL registered Assets, re-computing the Certificate hash from the same inputs SHALL produce the same hash value (deterministic hash invariant).
5. THE System SHALL provide a public certificate verification page accessible without authentication, where a user can enter a transaction hash and view the corresponding Certificate.

### Requirement 7: Admin Analytics Panel

**User Story:** As an Admin, I want to search, filter, sort, and export incident data, so that I can generate reports and track resolution progress.

#### Acceptance Criteria

1. THE Admin Analytics Panel SHALL display all Incidents in a paginated table with a default page size of 25 rows.
2. THE System SHALL allow Admins to filter Incidents by: date range, source type, Match_Score range, asset name, and resolution status.
3. THE System SHALL allow Admins to sort the Incidents table by any column in ascending or descending order.
4. WHEN an Admin applies filters, THE System SHALL update the Incidents table within 1 second.
5. THE System SHALL allow Admins to export the currently filtered Incident list as a CSV file containing all displayed columns.
6. THE System SHALL allow Admins to export the currently filtered Incident list as a PDF report containing a summary header, filter parameters, and the Incidents table.
7. THE System SHALL allow Admins to update the resolution status of an Incident to one of: Open, Under Review, Resolved, or Dismissed.
8. WHEN an Incident resolution status is updated, THE System SHALL record the update timestamp and the previous status value.

### Requirement 9: Scalability and Performance Architecture

**User Story:** As a platform architect, I want the system to be structured for scalability, so that it can support thousands of assets and concurrent scanning jobs in production.

#### Acceptance Criteria

1. THE System SHALL implement an asynchronous Scan_Job queue using a task queue pattern (e.g., Celery with Redis, or equivalent) to decouple scanning from the API request cycle.
2. THE System SHALL use the FAISS_Index for all similarity queries, avoiding full sequential scans of the Embeddings table.
3. THE System SHALL expose all backend functionality through a versioned REST API (prefix: /api/v1/).
4. THE System SHALL use a modular directory structure separating concerns into: asset registration, fingerprinting, scanning, incidents, analytics, and authentication modules.
5. WHEN the FAISS_Index contains 10,000 or more Embeddings, THE System SHALL return similarity query results within 500 milliseconds.
6. THE System SHALL store all persistent data (assets, incidents, certificates, job statuses) in a relational or document database with indexed query fields.

### Requirement 11: Guided Demo Mode

**User Story:** As a judge or investor, I want a one-click guided demo flow, so that I can experience the full platform capability in a structured walkthrough without manual setup.

#### Acceptance Criteria

1. THE System SHALL provide a Guided_Demo_Mode accessible from the main navigation that preloads a set of sample official Assets including at least one IPL highlight video clip.
2. WHEN Guided_Demo_Mode is activated, THE System SHALL auto-trigger an infringement detection simulation without requiring manual input from the user.

4. WHEN each step of the Guided_Demo_Mode scenario completes, THE System SHALL display a contextual explanation panel describing what occurred in that step.

### Requirement 12: User Roles and Access Control

**User Story:** As a platform administrator, I want role-based access control, so that each user type sees only the features and data relevant to their responsibilities.

#### Acceptance Criteria

1. THE System SHALL define the following roles: Admin, Sports_League_Manager, Broadcaster_Analyst, and Compliance_Officer.
2. THE System SHALL enforce the following access rules:
   - Admin: full access to all platform features and data.
   - Sports_League_Manager: manage Assets belonging to their own Organization and view Incidents related to those Assets.
   - Broadcaster_Analyst: read-only access to Incidents and analytics; no Asset creation or modification.
   - Compliance_Officer: view Incidents, manage takedown actions, and export reports; no Asset creation or modification.
3. THE System SHALL use JWT authentication for all protected API routes, rejecting requests without a valid token with an HTTP 401 response.
4. WHEN a JWT token expires, THE System SHALL issue a new access token using a valid refresh token without requiring the user to re-enter credentials.
5. IF a user attempts to access a resource outside their role's permissions, THEN THE System SHALL return an HTTP 403 response with a descriptive error message.
6. THE System SHALL render a role-specific dashboard on login, displaying only the features and panels relevant to the authenticated user's role.
7. WHEN a user's role is changed by an Admin, THE System SHALL apply the new permissions on the user's next authenticated request without requiring a platform restart.

### Requirement 13: Takedown and Enforcement Module

**User Story:** As a Compliance Officer, I want to manage takedown actions for detected infringements, so that I can enforce rights and track enforcement progress through to resolution.

#### Acceptance Criteria

1. WHEN an Incident is selected, THE System SHALL provide an option to generate a DMCA notice draft pre-populated with: Incident ID, matched Asset title, Organization name, suspicious upload URL, detection timestamp, and Match_Score.
2. WHEN a DMCA notice draft is generated, THE System SHALL display it in an editable text panel before submission.
3. THE System SHALL allow a Compliance_Officer to mark a takedown as requested, recording the requesting user ID and a UTC timestamp.
4. THE System SHALL support escalating an Incident's legal severity to one of four levels: Low, Medium, High, or Critical.
5. THE System SHALL track each Incident's takedown status through the following workflow states in order: Draft → Submitted → Acknowledged → Resolved.
6. WHEN a takedown status transitions to a new state, THE System SHALL record the transition timestamp and the user who triggered the transition.
7. IF a takedown status transition is attempted out of the defined workflow order, THEN THE System SHALL reject the transition and return a descriptive error message.
8. THE System SHALL display the current takedown status and full status history on the Incident detail page.

### Requirement 14: External API Integration Readiness

**User Story:** As a platform architect, I want the system architecture to support future integrations with external rights management platforms, so that enforcement actions can be automated across major content distribution channels.

#### Acceptance Criteria

1. THE System SHALL implement an integration adapter layer that defines a standard interface for connecting to external platforms including: YouTube Content ID, Meta Rights Manager, Twitter/X media monitoring, and Broadcast CDN feeds.
2. THE System SHALL expose webhook endpoints that accept HTTP POST callbacks from external systems, validating each callback with a shared secret signature.
3. WHEN a webhook callback is received and validated, THE System SHALL process the payload and update the corresponding Incident or Asset record within 10 seconds.
4. IF a webhook callback fails signature validation, THEN THE System SHALL reject the request with an HTTP 401 response and log the attempt.
5. THE System SHALL include an in-app README page that documents all integration points, webhook payload schemas, and the adapter interface contract.
6. THE System SHALL store integration configuration (endpoint URLs, credentials placeholders, enabled flags) in environment variables, not in source code.

### Requirement 15: Security Layer

**User Story:** As a platform administrator, I want comprehensive security controls, so that the platform protects sensitive asset data and resists common attack vectors.

#### Acceptance Criteria

1. THE System SHALL authenticate all protected API routes using JWT access tokens with a configurable expiry (default: 15 minutes) and refresh tokens with a configurable expiry (default: 7 days).
2. THE System SHALL store all Asset metadata at rest using encryption, ensuring plaintext metadata is not directly readable from the database storage layer.
3. WHEN a file upload is received, THE System SHALL validate the file type by inspecting the file's MIME type and magic bytes, rejecting files where the declared extension does not match the detected MIME type.
4. WHEN a file upload is received, THE System SHALL reject files that exceed the configured maximum size limit and return a descriptive error message.
5. THE System SHALL apply rate limiting to all API endpoints, with configurable per-endpoint request limits (default: 100 requests per minute per authenticated user).
6. WHEN a client exceeds the rate limit for an endpoint, THE System SHALL return an HTTP 429 response including a Retry-After header indicating when the client may retry.
7. THE System SHALL sanitize all user-supplied string inputs before persisting them to the database to prevent injection attacks.

### Requirement 16: Deployment Configuration

**User Story:** As a developer, I want the platform to be fully containerized with a local development compose file, so that I can run the entire stack locally and deploy each service independently to cloud providers.

#### Acceptance Criteria

1. THE System SHALL provide a Dockerfile for each of the following services: frontend, backend API, AI worker, and database.
2. THE System SHALL provide a docker-compose.yml file that orchestrates all four services for local development, including environment variable configuration and inter-service networking.
3. WHEN docker-compose up is executed from the project root, THE System SHALL start all services and make the application accessible on localhost within 60 seconds.
4. THE System SHALL configure the frontend service to be deployable to Vercel by including a vercel.json configuration file with the correct build and routing settings.
5. THE System SHALL configure the backend API service to be deployable to Render, AWS, or GCP by exposing a single HTTP port and reading all configuration from environment variables.
6. THE System SHALL containerize the AI worker as a separate service that can be scaled independently from the backend API service.
7. THE System SHALL include a .env.example file documenting all required environment variables for each service.

### Requirement 18: Performance Metrics Dashboard

**User Story:** As an Admin, I want to monitor the detection system's performance metrics in real time, so that I can assess accuracy and identify degradation.

#### Acceptance Criteria

1. THE Performance_Metrics_Dashboard SHALL display the following metrics: detection latency in milliseconds, precision score, recall score, and false positive rate.
2. THE Performance_Metrics_Dashboard SHALL display each metric for three selectable time windows: last hour, last 24 hours, and last 7 days.
3. WHEN a new Scan_Job completes, THE System SHALL update the Performance_Metrics_Dashboard metrics within 10 seconds without requiring a page refresh.
4. THE Performance_Metrics_Dashboard SHALL display each metric as both a numeric value and a trend indicator (up, down, or stable) compared to the previous equivalent time window.
5. THE System SHALL compute detection latency as the elapsed time in milliseconds between Scan_Job dispatch and Incident creation.
6. IF no Scan_Jobs have completed within the selected time window, THEN THE Performance_Metrics_Dashboard SHALL display a "No data for this period" message for the affected metrics.

### Requirement 21: Real-Time Event Streaming

**User Story:** As an Admin, I want live infringement alerts pushed to my dashboard over a persistent connection, so that I receive instant notifications without relying on polling.

#### Acceptance Criteria

1. THE System SHALL implement a WebSocket server (using Socket.IO or native WebSocket) that maintains persistent connections with authenticated dashboard clients.
2. WHEN a new Incident is created, THE System SHALL push a real-time event to all connected clients subscribed to the relevant Organization's incident channel within 2 seconds of Incident creation.
3. THE Alert_Dashboard SHALL update the live incident feed and summary statistics in response to received WebSocket events without requiring a page reload or HTTP poll.
4. WHEN a WebSocket connection is established, THE System SHALL authenticate the connection using the client's JWT token, rejecting unauthenticated connections.
5. IF a WebSocket connection is interrupted, THEN THE System SHALL attempt to reconnect automatically with exponential backoff, with a maximum retry interval of 30 seconds.
6. THE System SHALL support at least 100 concurrent WebSocket connections without degrading real-time event delivery latency beyond 5 seconds.
7. WHILE a WebSocket connection is active, THE System SHALL send a heartbeat ping every 30 seconds to detect and close stale connections.
